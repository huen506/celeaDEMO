## 二、技術決策紀錄

*每個「為什麼這樣決定」都記在這裡。已推翻的決定標記 [已變更] 並附新決定，不刪除。*

### 2026-04-15

- 【架構】採用模組化設計，各系統獨立運作，透過 EventManager 溝通，不直接互相呼叫。原因：降低耦合，後期修改單一系統不影響其他系統。

- 【命名】場景管理系統命名為 CeleaSceneManager，而非 SceneManager。原因：Unity 內建有 UnityEngine.SceneManagement.SceneManager，直接使用 SceneManager 會產生命名衝突。

- 【單例模式】EventManager、UIManager、CeleaSceneManager 均採用 Singleton + DontDestroyOnLoad。原因：這三個系統需要在整個遊戲生命週期中持續存在，且全域只需要一份。

- 【事件資料格式】EventData 採用 Dictionary<string, object> 的彈性格式，而非為每種事件建立專用類別。原因：減少類別數量，保持擴充彈性，新增事件不需要新增資料類別。

- 【場景狀態讀取失敗處理】讀取失敗時還原初始狀態，不報錯、不中斷遊戲。原因：開發初期 Bug 多，不能讓每次讀取失敗都中斷遊戲，影響開發流程。

- 【UI 劇情模式】劇情觸發時插圖與對話框同時出現，對話框固定在畫面下方。原因：對話框自然遮住角色立繪下半身，解決立繪高度不一致可能被截斷的問題。

- 【立繪設計】劇情模式最多四個立繪位置，彈性配置。原因：場景角色配置不固定，預留空間待後續決定。

### 2026-04-15（第二批補充）

- 【對話系統三層拆分】DialogueManager 拆成三層：接口層（DialogueManager）、核心層（DialogueStateMachine + DialogueFlowController）、功能層。原因：防止 DialogueManager 承擔過多責任變成肥大系統，企劃階段拆分成本為零，拖到程式碼完成後再拆代價太高。

- 【JSON 格式選型】對話資料 JSON 使用陣列格式儲存 segments，而非 Dictionary 格式。原因：Unity 內建 JsonUtility 不支援 Dictionary，改用陣列格式可直接用 JsonUtility 解析，不需安裝第三方套件。

- 【StepCounter 事件來源】步數計數由 PlayerController 在每次節點移動時發出 ON_PLAYER_STEP 事件，StepCounter 監聽累計。原因：明確定義事件來源，避免 StepCounter 不知道要聽誰的訊號。

- 【DialogueSaveData 移除】儲存系統不儲存對話進度（移除 DialogueSaveData）。原因：存檔只發生在劇情外的自由行動狀態，此時對話已結束，不存在進行中的段落進度需要儲存。

- 【儲存接口統一】各系統實作 CaptureState() 和 RestoreState() 兩個接口，SaveManager 只透過這兩個接口存取資料。原因：SaveManager 不直接碰各系統內部變數，新增系統時只要實作接口，存檔自動支援。

- 【善惡光譜消極選項行為】善第二階段選消極降低光譜；惡第二階段選消極提升光譜。邏輯在 MoralSpectrum 內部處理。原因：同一個「消極」選項在不同階段效果相反，必須在系統內部處理，不能依賴外部判斷。

### 2026-04-17（第三批補充）

- 【EventManager 補強】加入 `ENABLE_EVENT_LOG` 開關，預設 true，Publish 時印出 `[Event] 事件名稱`。原因：第三批事件數量達 20+ 個，蜘蛛網式依賴結構若無 log 輔助，除錯成本過高。

- 【HiddenQuest × DialogueLog 解耦】HiddenQuestManager 不直接呼叫 DialogueLogManager，改由 DialogueLogManager 自己監聽 ON_HIDDEN_QUEST_TRIGGERED 後找紀錄標記。原因：entryId 是執行時生成，直接呼叫有時序風險，事件式設計消除跨系統依賴。找不到對應紀錄時靜默跳過，不影響任務流程。

- 【NoticeBoard 狀態機】委託狀態改用 QuestState enum（Unavailable / Available / Active / Completed / Failed），步驟狀態改用 StepState enum（Locked / Revealed / Completed），所有轉移只能透過 NoticeBoardManager 的方法觸發，外部不允許直接賦值。原因：NoticeBoard 是有狀態、有流程、有時間依賴的流程型系統，bool flags 堆疊無法清楚表達轉移規則，技術債過重。

- 【SaveManager 還原順序寫死】RestoreState 固定順序：Time → Flag → HiddenQuest → NoticeBoard → DialogueLog → Gallery → Scene。硬編碼在 SaveManager.RestoreState 方法內，不透過事件，不讓各系統自己決定還原時機。原因：各系統還原時有跨系統依賴（Quest 需要 Flag 的好感度、NoticeBoard 需要 Time 的天數），順序錯誤會產生難以追蹤的 Bug。

### 2026-04-18（第四批補充）

- 【BattleStatsBuilder 鎖定機制】戰鬥中 isLocked = true，禁止重新組裝數值。收到 ON_COMBAT_START 鎖定，ON_COMBAT_END 解鎖。原因：防止好感度變動或魔核修改在戰鬥中途改變數值，造成數值跳動。

- 【CombatUnit 唯讀引用】CombatUnit 對 InjuryData 與 ResonanceData 只持有唯讀引用（private set），唯一賦值入口為 Initialize()。原因：確保傷勢與崩鳴的資料主權分別由 InjuryManager 與 ResonanceManager 持有，CombatUnit 不得自行修改。

- 【ResonanceData 初始化為零】每場戰鬥開始時崩鳴值一律初始化為 0，不繼承任何前一場數值。原因：崩鳴是戰鬥內即時狀態，不是跨場景資源。

- 【傭兵死亡同步清除】InjuryManager 監聽 ON_MERCENARY_DIED，收到後將該單位 InjuryData 標記為 Terminated 並從存檔資料中排除。原因：防止 isAlive = false 但 InjuryData 仍殘留「瀕死」的不一致狀態污染存檔。

- 【CoreModuleData 使用次數封裝】usageCount 改為 private int _usageCount，存取路徑唯一：寫入透過 internal IncrementUsage()，讀取透過 IsActive 布林判斷，不暴露數值。原因：紅色魔核次數絕對不能對玩家顯示，從資料層封死。

- 【MoralSpectrum 門檻修正】善惡第一門檻從 20 修正為 25，與 GrowthManager 對齊。確認門檻：-60 / -25 / +25 / +60。原因：驗收時發現兩個系統使用不同門檻值，道德值 20–24 的玩家光譜顯示善第一但抽中性池，修正為一致。

- 【場景 scoped 系統】CombatManager、ResonanceManager、BattleStatsBuilder 不採用 DontDestroyOnLoad，戰鬥結束後隨場景銷毀。原因：這三個系統是戰鬥 runtime，不需要跨場景存活，避免 Singleton 濫用。

- 【版本控制】採用 Git 進行版本控制。第一個 commit 於四批系統骨架全部驗收通過後建立，以此為乾淨存檔點。.gitignore 排除 Library/、Temp/、Obj/、Build/、Logs/、UserSettings/、.vscode/、.claude/ 等不需要納管的資料夾。後續每次重大進展後執行 git add . 與 git commit。原因：四批驗收全過、零編譯錯誤是最適合的第一個存檔點，確保版本歷史從乾淨狀態開始。

- 【SaveManager 還原順序擴充（第四批）】RestoreState 固定順序更新為：Time → Flag → Injury → Growth → CoreModule → Mercenary → HiddenQuest → NoticeBoard → DialogueLog → Gallery → Scene。原因：第四批新增四個持久系統，Injury 依賴角色身分（Flag 後）、Growth 依賴好感度（Flag 後）、CoreModule 依賴線路資訊（Growth 後）、Mercenary 依賴傷勢資料（Injury 後）。
